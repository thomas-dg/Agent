## 🎯 项目简介

**Coder** 是一个 AI 编码 Agent 的 **提示词工程（Prompt Engineering）** 项目。它不是一个可运行的程序，而是一套完整的 Agent 定义与评测体系，用于指导 LLM 生成高质量的 Go 代码。

核心产出是一个 **Go 代码重构 Agent**，能够：

1. 📥 接收用户的重构需求（方案文档、源代码、流程图）
2. 🏗 基于 **Clean Architecture** 进行架构重构
3. 📝 输出符合 **Go 语言设计哲学**（简洁、显式、组合）的生产级代码
4. ✅ 通过迭代评测不断提升代码生成质量

## 📦 模块详解

### 1. 重构 Agent 系统提示词（`Refactoring/system_prompt.md`）

这是 Agent 的核心定义文件，采用结构化的 Prompt 设计模式：

```
┌─────────────────────────────────────────────────────────┐
│                   System Prompt 结构                     │
├─────────────────────────────────────────────────────────┤
│  Role    │ 专业代码重构专家 + Go 语言专家                 │
├──────────┼──────────────────────────────────────────────┤
│  Goals   │ • 完成代码重构                                │
│          │ • 确保符合编程规范                              │
│          │ • 体现 Go 设计哲学（简单/显式/组合/并发/工程）   │
├──────────┼──────────────────────────────────────────────┤
│  Rules   │ • 应用 Clean Architecture 四层架构             │
│          │ • 代码简洁，避免过度设计                        │
│          │ • 遵循 Go 最佳实践（引用知识库）                 │
│          │ • 遵循标准化目录结构规范                        │
├──────────┼──────────────────────────────────────────────┤
│  Skills  │ 接收重构信息 → 执行重构 → 反馈结果              │
├──────────┼──────────────────────────────────────────────┤
│ Workflow │ 开场 → 分析 → 确认 → 重构 → 检查 → 编译 → 反馈 │
└──────────┴──────────────────────────────────────────────┘
```

**架构规范**：Agent 生成的代码严格遵循 Clean Architecture：

```
目标代码目录/
├── doc/                    # 文档
├── tests/                  # 集成测试
└── src/
    ├── entity/             # 领域模型（纯数据结构 + 常量）
    ├── logic/              # 业务逻辑（核心用例）
    ├── repo/               # 数据访问层（外部依赖封装）
    └── utils/              # 工具函数
```

### 2. Go 最佳实践知识库（`knowledge/go_best_practices.md`）

一份涵盖 **8 大章节** 的 Go 语言编码规范，作为 Agent 生成代码时的参考知识库：

| 章节 | 核心要点 |
|:---|:---|
| **核心指导原则** | 简洁优先、可读性至上、生产力导向 |
| **标识符与注释** | 清晰命名 > 简短命名、公共符号必须文档化 |
| **包与项目设计** | 少而大的包、main 包最小化、零值有用 |
| **API 设计** | 避免同类型多参数、默认用例优先、接口隔离 |
| **错误处理** | 消除错误 > 处理错误、仅处理一次、errors.Is/As |
| **并发编程** | goroutine 管理、并发留给调用者、errgroup |
| **Go 1.24+ 特性** | 泛型优化、runtime.AddCleanup、go mod tool |
| **禁用/慎用规则** | 禁用 init()、慎用 unsafe、避免深度嵌套 |

每条规则都配有正反对比的代码示例，确保 Agent 能准确理解并应用。

### 3. 评测体系（`Refactoring/evaluating/`）

- **user_prompt.md**：标准化的评测输入模板，模拟用户真实的重构请求
- **case.md**：真实代码评测案例，包含两种不同实现的对比分析，用于评判 Agent 生成代码的质量水平

## 🔄 迭代方法论

从 `CHANGELOG.md` 可以清晰看到**数据驱动的 Prompt 优化过程**：

### v0.0.1 → v0.0.2 演进路线

```
┌─────────────────────────────────────────────────────────────┐
│                    v0.0.1 评测发现                           │
├─────────────────────────────────────────────────────────────┤
│  ❌ 问题 1: 代码有 Java 语言的影子                           │
│  ✅ 解决: Role 增加 "Go 语言专家"                            │
│          Goals 增加 Go 设计哲学要求                          │
│                                                             │
│  ❌ 问题 2: 部分实现过于复杂（工厂模式滥用）                   │
│  ✅ 解决: Rules 增加 "避免过度设计"                           │
│                                                             │
│  ❌ 问题 3: 代码规范类问题（常量、errgroup 用法）              │
│  ✅ 解决: 整理 Go 最佳实践知识库 + 引入 RAG                   │
│                                                             │
│  ❌ 问题 4: 目录结构与规范有差异                              │
│  ✅ 解决: Prompt 中增加详细目录规范                           │
│                                                             │
│  ❌ 问题 5: 补偿队列实现过于复杂                              │
│  ✅ 解决: 改用同步发送 Kafka                                 │
├─────────────────────────────────────────────────────────────┤
│                    v0.0.2 改进成果                           │
│  • System Prompt 更完善                                     │
│  • 新增 Go 最佳实践知识库                                    │
│  • 代码生成质量显著提升                                      │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：
- Prompt 优化不是"一次写好"，而是通过 **评测 → 分析 → 改进** 的循环迭代
- 知识库（RAG）是解决领域规范类问题的有效手段
- 精确的 Role 和 Rules 定义能显著影响代码风格和质量
