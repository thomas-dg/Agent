# Changelog

## v0.0.1
### Summary
agent第一版

### Evaluating
#### 优点
1. 代码看起来质量挺高
2. 假定了推荐和内容处理两个场景，定义了处理流程

#### 不足
1. 目录结构与规范有一些差异
   解决：Prompt中增加目录规范约束

2. 流程图画的有歧义，导致每个处理流(goroutine)实现不一样。但其实，处理流程是一样，实现不一样的处理流程，反而增加了复杂度
   解决：完善流程图，更新流程说明文档

3. 补偿队列实现有些复杂（多worker+chan），我们的场景可以使用同步发送kafka实现，降低复杂度
   解决：更新流程说明文档，补充补偿队列实现说明

4. 监控逻辑耦合在了处理路程中，期望通过AOP实现
   解决：文档中补充监控逻辑实现


## v0.0.2 
### Summary
- 修复0.0.1评测发现的不足
- 补充目录规范说明

### Evaluating
#### 优点

#### 缺点
1. 代码发现了不少java语言的影子，不是go语言的风格
   解决：role增加“go语言专家”，Goals增加go语言设计哲学说明，skills增加代码风格相关说明

2. 部分实现过于复杂，可以用更简单的实现
   解决：前面第一点，应该解决了一部分；同时在Rules增加“代码尽量简洁，避免过度设计”

3. 代码规范类，比如插件名、阶段名可以定义为常量；启动多个goroutine切感知error的情况下，应该使用errgroup
   解决：整理go语言最佳实践，作为知识库，以便agent查询；同时，Rules新增要求代码需参考最佳实践


## v0.0.3 
### Summary
- 修复0.0.2评测发现的不足

### Evaluating
#### 优点

#### 缺点
1. 总是很多的Manager，看着像是java的语言风格
   解决：go语言最佳实践中增加语言风格约束，并在Prompt rules中增加“坚决反对将 Java、C# 等面向对象语言的设计模式和习惯带入 Go 代码中”

2. 打日志使用 git.code.oa.com/trpc-go/trpc-go/log，而不是go的基础log
   解决：整理标准库使用规范，作为最佳实践的一部分

3. 会自作聪明做一些事情，比如我已经定义了接口，但是他会按照自己的理解修改接口定义
   解决：Prompt rules中增加“人类代码拥有绝对优先权”规则
